# Секреты и безопасность

## Почему не хранят «зашифрованные» секреты в Git

Next.js при запуске читает переменные окружения в открытом виде. Ciphertext в репозитории всё равно требует ключа расшифровки — хранить ключ рядом бессмысленно. Нормальная схема:

- Секреты только в **`.env.local`** (не в Git) и в **панели хостинга** (Vercel и т.д.).
- В Git — только **`.env.example`** без реальных значений.

Подробнее: таблица переменных, ротация токенов, BitLocker — ниже.

## Что считать секретом

- **Секреты:** `UPSTASH_REDIS_REST_TOKEN`, `MAKE_WEBHOOK_SECRET`, `ADMIN_PASSWORD`, `ADMIN_SESSION_SECRET`, `YANDEX_METRIKA_OAUTH_TOKEN`.
- **`NEXT_PUBLIC_*`** попадают в браузер — это не «скрытые» ключи, но в `.env.example` их можно оставлять пустыми.

## Утечка в чат / скрин

1. Upstash: консоль — rotate token.
2. Make: новый `MAKE_WEBHOOK_SECRET` в `.env.local`, на хостинге и в Make.
3. Админка: сменить пароль и при необходимости `ADMIN_SESSION_SECRET`.

## Шифрование на диске (Windows)

`.env.local` остаётся текстовым файлом. Дополнительно: **BitLocker** на диске или шифрование папки пользователя.

## Проверка перед коммитом

Из папки `marsmedia`:

```bash
npm run secrets:check
```

Скрипт ищет типичные паттерны утечек (длинные base64-подобные токены). Возможны ложные срабатывания.

## Хостинг

Секреты только в **Environment Variables** провайдера, не в репозитории.
